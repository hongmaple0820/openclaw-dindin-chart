# 🔄 长任务处理机制

> 避免 AI 在执行长任务时"卡住"，让用户等待过久无响应。

## 问题场景

- 连续生成多个大文件
- 执行复杂的代码分析
- 批量处理数据
- 长时间的 API 调用

**后果**：用户发消息没有反应，以为 AI 宕机了。

---

## 解决方案：分步反馈 + Sub-Agent

### 方案 1：分步反馈（简单任务）

执行长任务时，**每完成一步就给用户反馈**：

```
用户：帮我写 6 份设计文档

我：好的，开始生成 6 份设计文档，预计需要 2-3 分钟...

[生成第 1 份]
我：✅ 架构设计完成 (1/6)

[生成第 2 份]
我：✅ 数据库设计完成 (2/6)

...

我：✅ 全部完成！文档已保存到 docs/ 目录。
```

**实现方式**：
- 不要一次性调用多个 write 工具
- 每完成一个文件，先回复用户，再继续下一个
- 或者在单次回复中分批调用，中间穿插状态说明

### 方案 2：Sub-Agent（复杂任务）

把长任务交给后台子 Agent，主会话保持响应：

```
用户：帮我分析这个项目的代码

我：这个任务比较耗时，我派一个子 Agent 去处理，完成后会通知你。
   主会话保持响应，有其他问题可以继续问我。

[调用 sessions_spawn]
sessions_spawn({
  task: "分析 /path/to/project 的代码结构，生成分析报告",
  label: "code-analysis",
  runTimeoutSeconds: 300
})

[子 Agent 完成后自动通知]
```

**适用场景**：
- 代码分析
- 大量文件处理
- 需要多轮 API 调用的任务
- 预计超过 1 分钟的任务

---

## 判断标准

| 任务类型 | 处理方式 |
|:---------|:---------|
| 写 1-2 个文件 | 直接执行 |
| 写 3-5 个文件 | 分步反馈 |
| 写 6+ 个文件 | 考虑 Sub-Agent |
| 代码分析 | Sub-Agent |
| 批量数据处理 | Sub-Agent |
| 需要等待外部 API | Sub-Agent |

---

## 最佳实践

### 1. 开始前告知

```
我：这个任务需要一些时间，我会分步反馈进度...
```

### 2. 过程中反馈

```
我：⏳ 正在处理第 3/10 项...
```

### 3. 完成后总结

```
我：✅ 全部完成！
- 生成了 6 份文档
- 总计 50KB
- 已提交到 Git
```

### 4. 异常时主动说明

```
我：⚠️ 第 4 项处理失败，原因是...
继续处理剩余项目...
```

---

## 记住

- **用户的等待时间 > 30 秒** → 需要中间反馈
- **用户的等待时间 > 2 分钟** → 考虑 Sub-Agent
- **永远不要让用户在没有任何反馈的情况下等待超过 1 分钟**
